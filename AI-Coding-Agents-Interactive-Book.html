Chapter 01

The World Before Agents

Why do AI coding agents exist? What were we doing before them? Understanding the history helps you understand the present — and what agents are actually good at.

The Problem Agents Solve

Software development has always required you to be the translator between intent and implementation at every step. You knew what you wanted. The computer could help you type it faster. But the cognitive work — navigating code, planning changes, writing, testing, debugging — was entirely on you.

AI coding agents automate this entire cycle. They can read files, plan changes, write code, run tests, read error messages, fix failures, and loop until everything works. No single step is done better than a skilled developer. But the entire loop runs at machine speed, without losing focus.

The Three Eras

⌨️

Era 1: Deterministic (1996–2018)

IntelliSense, IDE refactoring, scaffolding tools, linters. The computer knew the rules of the language. Zero intelligence, maximum reliability.

🧠

Era 2: Neural Completion (2018–2022)

TabNine, Kite, Codex, Copilot. The computer had seen millions of code files. It could predict what you'd type next. Pattern-based, not understanding-based.

🤖

Era 3: Agentic (2022–Now)

ChatGPT, Devin, Claude Code, Cursor Agent. The computer can understand what you want, write the code, test it, and fix its own mistakes autonomously.

Key Insight

Agents didn't emerge from a single invention. They required three threads converging: models got good enough at code (Codex→GPT-4→Claude), tool-use APIs became first-class (function calling, June 2023), and the ReAct loop pattern showed how to iterate (Oct 2022).

Chapter 02

The Agent Loop — How Agents Think

Every coding agent — Claude Code, Cursor, Devin, Codex — runs the same fundamental loop. Understanding it is like understanding how an engine works before driving a car.

The Core Pattern: ReAct

Published October 2022 by Shunyu Yao et al., the ReAct pattern interleaves reasoning and action. Before ReAct, models either reasoned internally (Chain-of-Thought, prone to hallucination) or acted without reasoning (prone to aimless exploration). ReAct combines both:

The Entire Architecture in Code

The agent loop fits in under 20 lines of logic. Here it is — the same pattern used by Claude Code, Cursor, and Codex:

def agent_loop(query): messages = [{"role": "user", "content": query}] while True: response = client.messages.create( model=MODEL, system=SYSTEM, messages=messages, tools=TOOLS, ) messages.append({"role": "assistant", "content": response.content}) if response.stop_reason != "tool_use": return # Model finished — no more tool calls results = [] for block in response.content: if block.type == "tool_use": output = run_tool(block.name, block.input) results.append({"type": "tool_result", "tool_use_id": block.id, "content": output}) messages.append({"role": "user", "content": results})

That's it. The model generates text or tool calls. If text → done. If tool calls → execute them, feed results back, repeat. Every coding agent in existence is a variation of this loop.

Four Universal Components

🧠

Brain (LLM)

The model that generates reasoning and tool calls. Claude, GPT-4, etc.

📋

Instructions

System prompt + CLAUDE.md. Tells the model who it is and how to work.

🔧

Tools

Functions the model can call: read files, run commands, search, etc.

💾

Memory

The message history — every user message, assistant response, and tool result.

Why This Matters

The loop gives agents their most important capability: the ability to recover from mistakes. A single-shot model that writes buggy code is stuck. An agent that writes buggy code can run the tests, read the error, reason about the cause, and try again.

📚 Dig Deeper — ReAct Performance Numbers

On ALFWorld (text-based navigation): ReAct achieved 71% success vs 45% for action-only. On Fever (fact verification): 60.9 EM vs 56.3 for Chain-of-Thought. The best approach was ReAct → CoT-SC fallback: try the agent loop first, fall back to pure reasoning if the loop gets stuck.

Original paper: arXiv:2210.03629 · Simon Willison's implementation: Python ReAct pattern

Chapter 03

Tool Use & Function Calling

An LLM without tools is a brain in a jar. Tools give it hands. This chapter explains the exact mechanism — and why tool descriptions matter more than the model itself.

The Five-Step Tool Calling Protocol
Click each step to see the details
1
Define tools as JSON schemas

You describe each tool's name, purpose, and parameters. The model learns what it can do from these descriptions.

2
Send request with tools + message

The API receives the user's question alongside the tool definitions. Behind the scenes, tools become part of the prompt.

3
Model returns structured tool calls

Instead of free text, the model outputs JSON: which tool to call and with what arguments. Schema-enforced via constrained decoding.

4
YOUR code executes the function

The LLM never runs code directly. Your application receives the tool call, executes it, and captures the result.

5
Send result back, model continues

The tool result becomes a new message. The model reads it and either calls another tool or generates a final response.

The #1 Finding

Anthropic found that tool descriptions are "by far the most important factor in tool performance." Optimizing descriptions achieved a 40% decrease in task completion time. A description should be 3-4+ sentences, explain when to use the tool, what it returns, and what to watch out for.

MCP: The Universal Connector

The Model Context Protocol (Anthropic, Nov 2024) is "USB-C for AI." Before MCP: N models × M tools = N×M integrations. With MCP: each model implements it once, each tool implements it once. Adopted by OpenAI, Google, Cursor, Zed, and 60K+ projects.

🔧

Tools

Executable functions that change state. The model decides when to call them.

📄

Resources

Read-only data via URIs (file://, db://). The app decides when to include them.

📝

Prompts

Reusable templates. User-controlled, retrieved and customized on demand.

Five Composable Patterns

From Anthropic's canonical guide. The building blocks for every agent system. The golden rule: start simple, add complexity only when measurement proves it's needed.

Input → LLM 1 → ✓ Gate → LLM 2 → ✓ Gate → Output

What: Fixed sequence of LLM calls. Each step processes the previous output. Gates validate quality between steps.

When: Task decomposes into predictable subtasks. You want higher accuracy by making each call simpler.

Example: Generate code → validate syntax → generate tests → run tests → fix failures.

Risk: Cascade failures — bad output from step 1 poisons everything. Latency scales linearly.

Input → Router → billing? → Billing Handler
Input → Router → technical? → Tech Handler

What: Classify input, direct to specialized handler. Each handler optimized independently.

When: Distinct categories needing different prompts, tools, or even models.

Example: Simple queries → Haiku (fast/cheap), complex → Opus (powerful).

Risk: Misclassification sends inputs to wrong handler. Ambiguous inputs that span categories.

--- - Security ↘
Input → Quality → Merge
--- - Tests ↗

What: Multiple LLM calls run simultaneously. Results aggregated. Two variants: sectioning (split task) and voting (same task, multiple tries).

When: Genuinely independent subtasks. Or when you need reliability via majority vote.

Example: Review code security + quality + tests in parallel. Run guardrail alongside main response.

Risk: Results may be inconsistent. Aggregation logic must handle contradictions.

--- - Worker 1 ↘
Task → Orchestrator → Synthesize
--- - Worker 2 ↗

What: Central LLM dynamically plans subtasks, delegates to workers, synthesizes results. Subtasks are NOT predetermined.

When: You can't predict what subtasks are needed. Multi-file coding — the orchestrator analyzes the task first.

This is how Claude Code, Cursor Agent, and Devin work.

Risk: Vague delegation produces poor worker output. Workers may produce conflicting changes.

Task → Generator → Evaluator → Approved? → Output
...- ↑ No ←——————— feedback ——

What: One LLM generates, another evaluates and provides feedback, looping until quality criteria met.

When: Output demonstrably improves from iterative feedback. Translation, code optimization, creative writing.

Practical note: Diminishing returns after 2-3 iterations. Overly strict evaluators cause infinite loops.

🧪 You need to build an agent that handles both billing questions AND technical debugging. Which pattern?
Prompt Chaining — process billing then technical
Routing — classify the input, direct to a specialized handler
Parallelization — run both handlers simultaneously
Evaluator-Optimizer — iterate until the answer is good
Routing is correct because billing and technical support are distinct categories that benefit from different prompts, tools, and potentially different models. The router classifies first, then each handler operates independently with optimized context.
Chapter 05

Context Engineering

Prompt engineering was about crafting the right question. Context engineering is about curating the right information. It's the most impactful skill in the entire agent toolkit.

Andrej Karpathy's analogy: the LLM is a CPU. The context window is RAM. Your job is the operating system — loading exactly the right data for each task. Random data in RAM = bad performance. Random context in the window = bad agent output.

Three Retrieval Strategies

📦

Upfront

Load everything potentially relevant before inference. CLAUDE.md, RAG. Fast and predictable, but may waste space.

🔍

Just-in-Time

Agent uses grep/glob/read to explore on demand. Maximally efficient but costs tool calls. How Claude Code explores codebases.

⚡

Hybrid

CLAUDE.md (upfront) + filesystem tools (JIT). The recommended approach. All major agents use this.

CLAUDE.md Quick Reference

# MyProject ## WHY: Project Overview E-commerce API, 50K RPM. TypeScript strict mode. PostgreSQL via Prisma ORM. ## WHAT: Key Commands - `npm run dev` — Start dev server - `npm test` — Jest tests (needs local DB) - `npm run lint` — ESLint + Prettier ## HOW: Conventions - All handlers use zod for validation - Response shape: { data: T | null, error: string | null } - Never use `any` — use `unknown` and narrow ## Watch Out For - Tests use real local DB, NOT mocks - Payments module has its own rules (src/payments/CLAUDE.md)

Key rules: Keep under 150-200 instructions. Tell Claude how to find info, not the info itself. Add rules reactively when you notice repeated mistakes. Use WHY/WHAT/HOW structure.

Chapter 07

Practical Workflows

The day-to-day workflows that separate casual users from power users. Agents reward existing engineering discipline — testing, planning, documentation, version control.

🗺️

Waterfall in 15 Minutes

Interview → Spec → Plan → Implement. Ask the agent to interview you about requirements BEFORE coding. Saves hours of rework.

✅

Test-Driven Agentic Dev

Write tests first (in isolation, no peeking at implementation). Then implement against those tests. Reduced regressions by 70%.

🎤

The Interview Workflow

"Interview me with 8 questions about [feature], then create a spec." Forces hidden assumptions into the open. Developers report 40+ questions surfacing requirements they'd never have thought of.

💾

Git Checkpointing

Commit after each successful small change. Treat commits as "save points in a game." Granular rollback when things go sideways.

🌲

Multi-Worktree Parallel

Use git worktree add to run multiple agent sessions on different features simultaneously. True parallelism.

🧹

The /clear Discipline

Between unrelated tasks, reset context. Old task context is noise. If the next task doesn't build on the previous, clear.

Chapter 08

The Think Tool

A tool that does nothing — and improves performance by 54%.

The think tool is a no-op tool. It accepts a thought and returns it unchanged. No external action, no new information — just structured reflection space between tool calls. On τ-bench (airline domain), it improved performance from 0.332 to 0.584 — a 54% relative improvement.

🔮

Extended Thinking

Activates before the response. Deep pre-planning. Best for coding, math, architecture decisions.

💭

Think Tool

Activates during the response, between tool calls. Processing external info as it arrives. Best for sequential chains + policy-heavy tasks.

Chapter 09

Evaluating Agents

Traditional testing assumes determinism. Agents are non-deterministic. You need a different approach: evaluations (evals).

☑️

Deterministic Graders

Tests pass/fail, syntax checks, keyword matching. Fast, cheap, reproducible. Can only verify objective criteria.

🤖

LLM-as-Judge

Another LLM evaluates quality. 80%+ agreement with humans. Flexible but introduces its own biases (verbosity, position).

👤

Human Evaluation

The gold standard. Expensive and slow. Best used for calibrating automated graders.

pass@k vs. pass^k Calculator
Two metrics that look similar but measure completely different things

pass@k = at least ONE success in k tries (measures capability)

pass^k = ALL k trials succeed (measures consistency)

98.4%
pass@k (at least one success)
42.2%
pass^k (all must succeed)
Key Principle

Eval-driven development: Build evals before building features. Start with 20-50 tasks from real failures. Two categories: capability evals (climbing toward higher pass rates) and regression evals (maintaining 100%). When capability evals hit high pass rates, they graduate to the regression suite.

The Compound Error Problem

The most important chapter in this book. Every other concept exists to mitigate the problem described here.

Overall success rate (pn)
At 95% per-step accuracy, a 10-step task fails 40% of the time

This Explains Everything

PhenomenonCompound Error Explanation
The 70% ProblemAI gets 70% done (low n), but the remaining 30% requires many steps (high n) where errors compound
Productivity ParadoxSimple tasks (low n) speed up. Complex tasks (high n) slow down or break entirely
Multi-Agent TrapEach agent adds steps. More agents × more steps = exponential error growth
Expert developers slowerExperts tackle harder tasks (higher n) where compound errors dominate

Five Mitigation Strategies

📉

Reduce n

Fewer, more capable steps. One rich LLM call beats five thin ones.

📈

Increase p

Better models, better context, better tools. 95%→99% per-step transforms 10-step success from 60%→90%.

✅

Verify Each Step

TDD, type checking, linting. Catch errors immediately. TDD reduced regressions by 70%.

👤

Human-in-the-Loop

Human critics boost completion rates by ~30 percentage points. AI critics show marginal gains.

📦

Richer Context

Instead of 10 steps with thin context, use 3 steps with rich context. Context engineering reduces n.

Chapter 12

The Road Ahead

Five predictions for 2026–2028, and five enduring principles that will outlast any specific tool.

Five Enduring Principles

1

Simplicity Wins

Start with the simplest approach that could work. A single LLM call with good tools beats a complex multi-agent swarm in most cases. Add complexity only when measurement proves it's needed.

2

Context Is Everything

The quality of an agent's output is bounded by the quality of its context. Every token in the context window should earn its place. Invest in context engineering above all else.

3

Compound Errors Are the Fundamental Constraint

P(success) = pn. Design architectures to minimize sequential dependencies and maximize verification at each step.

4

Measure, Don't Assume

The METR study's 39-percentage-point perception gap proves intuitions about AI effectiveness are systematically wrong. Measure your own team's results.

5

Agents Amplify Engineering Discipline

Testing, planning, documentation, version control, code review — these practices become more valuable, not less, in an agent-assisted workflow. The agents reward the practices good engineers already follow.

The Bottom Line

The tools will change. The principles won't. The age of AI coding agents is just beginning. The foundations you've learned — the agent loop, tool use, composable patterns, context engineering, compound error mitigation — will serve you regardless of which specific tools dominate in two years.
