Chapter 1 Overview of ML Systems

Before building anything, understand what makes ML systems different from traditional software. ML systems aren't just models — they are complex, evolving systems where code is only a small fraction. The real challenge is everything around the model: data, infrastructure, monitoring, and people.

Core Insight

ML in production ≠ ML in research. Research optimizes for model performance on fixed benchmarks. Production optimizes for business value under constantly changing real-world conditions, latency constraints, and reliability requirements.

🧩

ML Systems vs Traditional Software

ML systems are partly code, partly data, partly learned behavior — making them fundamentally different to reason about.

In traditional software, you write explicit rules. In ML, the "rules" are learned from data. This means:

  • Bugs can be in your data, not just your code
  • Behavior changes when data changes — even without a code deploy
  • Testing is harder because you can't enumerate all possible inputs
  • Versioning must cover code, data, AND models

Think of it as building a car where the engine redesigns itself every night based on the roads it drove on yesterday.

▼ Click to expand
🎯

When to Use ML

ML is powerful but not always the right tool. Use it when the problem is learnable, has data, and patterns are too complex for explicit rules.

ML is a good fit when:

  • The system needs to learn patterns (recommendations, NLP, vision)
  • Patterns are complex and can't be hand-coded
  • Data is available or can be collected
  • Predictions don't need to be 100% correct every time
  • Patterns change over time (fraud, user preferences)

ML is NOT a good fit when: deterministic logic works, you have no data, errors are catastrophically expensive, or simple heuristics solve it.

▼ Click to expand
👥

Stakeholders & Requirements

ML systems have many stakeholders with competing requirements: ML engineers want accuracy, business wants ROI, users want speed.

Each stakeholder cares about different things:

  • ML Engineers — model performance, interesting problems
  • Business/Product — revenue impact, cost, time to market
  • Users — latency, relevance, fairness, privacy
  • DevOps/Platform — reliability, scalability, maintainability

Your job is to translate between these worlds. A 2% accuracy improvement means nothing if it doubles latency and costs $500K/year in compute.

▼ Click to expand
📊

The Real ML System Breakdown

Google's famous figure: ML code is just 5% of a real ML system. The rest is data, serving, monitoring, and infrastructure.

A production ML system includes:

  • Data collection & verification
  • Feature extraction & storage
  • Model training & validation
  • Serving infrastructure
  • Monitoring & alerting
  • Configuration & orchestration
  • Resource management

The model itself is a small box in a much larger architecture diagram. If you only focus on the model, you're missing 95% of the problem.

▼ Click to expand
🧠 Memory Anchor

"ML System = Iceberg"

The model is the tip above water. Below the surface: data pipelines, feature stores, serving infra, monitoring, retraining loops, config management. Most production incidents happen below the waterline.

🔬 Dig Deeper: ML System Failure Modes

+

Unlike traditional software bugs, ML system failures are often silent. The system continues to serve predictions, but they're wrong:

  • Data pipeline corruption — a column meaning changed upstream but nobody updated the feature
  • Training-serving skew — features computed differently at training vs inference time
  • Feedback loops — the model's predictions influence the data it later trains on, creating spirals
  • Concept drift — the real-world patterns changed but the model didn't
  • Undeclared consumers — someone built another system on your model's output and neither of you knew

Explore further: Read Google's "Hidden Technical Debt in Machine Learning Systems" paper. Study how companies like Netflix and Uber structure their ML platform teams.

Chapter 2 ML System Design & Objectives

Designing an ML system starts with framing the problem correctly. The way you frame your ML objective determines everything downstream — what data you need, what model you choose, and how you measure success.

Core Insight

Business objectives ≠ ML objectives. The business wants "increase revenue." You need to translate that into "predict click-through probability" or "rank items by purchase likelihood." This translation is the hardest and most important step.

The Four Requirements of ML Systems

⚡

1. Reliability

The system performs correctly even when things go wrong — hardware failures, unexpected inputs, data corruption.

Reliability means the system continues to work correctly at the desired level of performance, even when things don't go as planned. For ML this specifically means:

  • Graceful degradation under bad inputs
  • Fallback strategies when the model fails
  • No silent failures — monitor everything
  • The system should be wrong in predictable, bounded ways
▼ Click to expand
📈

2. Scalability

Can handle growing complexity — more data, more models, more users — without proportional increase in effort.

Scalability isn't just about traffic. In ML, it also means scaling:

  • Data — from GBs to TBs of training data
  • Model complexity — from logistic regression to ensembles to LLMs
  • Artifact management — hundreds of models, experiments, feature versions
  • Team — multiple teams shipping models independently

Resource scaling: auto-scaling inference servers, distributed training, efficient feature stores.

▼ Click to expand
🔧

3. Maintainability

Different contributors can work on the system productively. Code, data, and models are understandable and modifiable.

ML systems are notorious for becoming unmaintainable. Key practices:

  • Documentation — not just code, but data schemas, model cards, experiment logs
  • Modularity — separate data processing, training, serving, monitoring
  • Reproducibility — any past result should be reproducible
  • Clear ownership — who owns the model? the data? the feature pipeline?
▼ Click to expand
🔄

4. Adaptability

The system can adjust to changing data distributions and business requirements without a full redesign.

The world changes. Your ML system must change with it:

  • Data distribution shifts — user behavior evolves, seasonality, new markets
  • Business pivots — the optimization target changes
  • Regulatory changes — new fairness or privacy requirements

Design for change from day one: parameterize what changes, automate retraining, use feature flags, A/B test everything.

Reliable · Scalable · Maintainable · Adaptable

Remember: "Real Systems Must Adapt" — The four pillars of production ML system design. Every architecture decision should be evaluated against these four dimensions.

How to Frame: "Increase User Engagement"

This single business goal can be framed as multiple ML problems:

  • Classification — Will this user churn? (Binary)
  • Regression — How many minutes will this user spend? (Continuous)
  • Ranking — Which content should we show first? (Ordered list)
  • Multi-label — Which categories does this user like? (Multiple labels)

The right framing depends on what action you can take on the prediction and what data you have available.

🔬 Dig Deeper: Objective Functions & Decoupling

+

A critical skill is decoupling objectives. Many real problems have multiple objectives that conflict:

  • Relevance vs. Revenue — show what users want vs. what advertisers pay for
  • Accuracy vs. Fairness — maximize overall accuracy vs. equalize performance across groups
  • Precision vs. Recall — avoid false positives vs. avoid false negatives

Approaches:

  • Multi-objective optimization — Pareto frontiers, weighted loss functions
  • Cascading systems — first filter for quality, then rank for relevance, then re-rank for diversity
  • Constraint-based — maximize metric A subject to metric B ≥ threshold

Explore further: Study how Netflix decomposes recommendations into candidate generation → ranking → re-ranking. Look into multi-task learning and Pareto efficiency in ML.

Data Formats

Choosing between JSON, CSV, Parquet, Avro, Protobuf, and others determines how efficiently you can store and process data.

Row-oriented (CSV, JSON): good for writing and reading individual records. Best for transactional workloads and when you need all columns.

Column-oriented (Parquet, ORC): good for analytical queries on specific columns. Best for ML feature extraction — you rarely need all columns.

Binary formats (Protobuf, Avro): compact, fast, schema-enforced. Best for streaming and service-to-service communication.

Rule of thumb: Use Parquet for analytical/ML workloads. Use Protobuf or Avro for streaming. Use JSON only for human-readable configs.

▼ Click to expand
🏗️

Data Models

Relational, document, and graph models each suit different ML data patterns. Understanding trade-offs is key.

Relational (SQL) — structured, joins, ACID transactions. Great when your data has clear schema and relationships. Most ML feature pipelines start here.

Document (NoSQL) — flexible schema, nested structures. Good for unstructured data, user profiles with varying fields.

Graph — nodes and edges, relationship-first. Perfect for social networks, knowledge graphs, fraud detection (KùzuDB, Neo4j).

For ML, you often need denormalized data — flatten everything into wide feature tables. The data model affects how painful this is.

▼ Click to expand
🌊

Batch vs Stream Processing

Batch processes historical data in large chunks; stream processes data in real-time as it arrives. Most ML systems need both.

Batch processing (Spark, Dask):

  • Process large datasets periodically (hourly/daily)
  • Higher throughput, higher latency
  • Good for: training data prep, feature backfills, reporting

Stream processing (Kafka + Flink/Spark Streaming):

  • Process events as they arrive
  • Lower latency, more complex to build
  • Good for: real-time features, live fraud detection, online learning

Lambda Architecture: Run both batch and stream, combine results. Complex but powerful.

Kappa Architecture: Stream-only with replay capability. Simpler, gaining popularity.

▼ Click to expand
🔗

ETL vs ELT

Extract-Transform-Load vs Extract-Load-Transform — the order matters more than you think for ML pipelines.

ETL (traditional): Transform data before loading into the warehouse. Data arrives clean, but transformations are rigid — if you need different features, you rebuild the pipeline.

ELT (modern): Load raw data first, transform later. More flexible for ML because you can create new features from raw data without re-ingesting.

For ML, ELT is usually better because:

  • You don't know what features you'll need upfront
  • Raw data enables experimentation
  • Feature engineering is iterative
  • Compute is cheaper than re-ingestion
▼ Click to expand
📏 Rule of Thumb

"If your training data changes, your model changes." Treat your data pipeline with the same rigor as your code pipeline: version control, testing, CI/CD, monitoring, and rollback capabilities.

🔬 Dig Deeper: Data Storage for ML

+

Where your data lives affects everything:

  • Data Lake (S3, GCS) — raw, unstructured. Cheap, flexible, but slow queries. Good for storing everything.
  • Data Warehouse (BigQuery, Snowflake, Redshift) — structured, optimized for SQL. Good for feature extraction.
  • Data Lakehouse (Delta Lake, Iceberg, Hudi) — best of both. Raw storage + ACID transactions + fast queries. Increasingly the gold standard for ML.
  • Feature Store (Feast, Tecton) — purpose-built for ML features. Handles online/offline serving, versioning, lineage.
  • Vector Database (LanceDB, Pinecone, Weaviate) — for embeddings and similarity search. Essential for modern retrieval-augmented systems.

Explore further: Study the Data Mesh paradigm (domain-oriented data ownership). Learn about data contracts — formal agreements between producers and consumers about schema, quality, and SLAs.

Chapter 4 Training Data

The quality of your training data sets the ceiling for your model. Most ML project time is spent on data, not model architecture. Understanding sampling, labeling, class imbalance, and data augmentation is critical.

Core Insight

Models are only as good as their training data. If you're spending 80% of your time on model tuning and 20% on data, you have it backwards. Flip that ratio — better data almost always beats a better model.

🎲

Sampling Strategies

How you sample data from the real world into your training set determines what your model learns and what it misses.

Non-probability sampling (convenience, snowball, judgment) — easy but biased. Your model learns the biases of your sampling.

Simple random sampling — every item equally likely. Good baseline but can miss rare events.

Stratified sampling — ensure proportional representation of subgroups. Critical when some classes are rare.

Reservoir sampling — sample from a stream without knowing the total size. Key for streaming data.

Importance sampling — oversample rare/important events, apply weights to correct the bias.

▼ Click to expand
🏷️

Labeling

Getting labels is expensive and error-prone. The labeling strategy you choose can make or break your project timeline and model quality.

Hand labeling — accurate but slow and expensive. Subject to annotator disagreement.

Natural labels — labels generated by user behavior (clicks = positive, no click = negative). Free but noisy and have feedback delay.

Programmatic labeling (Snorkel) — use heuristics, rules, and weak classifiers to generate noisy labels. Then denoise with label models.

Semi-supervised — small labeled set + large unlabeled set. Self-training, co-training, label propagation.

Active learning — have the model identify which samples it's most uncertain about, then label only those. Maximum information per label.

Transfer learning — use a model pre-trained on a large dataset, fine-tune with small labeled data.

▼ Click to expand
⚖️

Class Imbalance

When 99% of emails are not spam and 1% are spam, naive models learn to always predict "not spam" — and get 99% accuracy.

Data-level approaches:

  • Oversampling minority class (SMOTE — creates synthetic examples)
  • Undersampling majority class (Tomek links, random undersampling)
  • Data augmentation — create more minority examples through transformations

Algorithm-level approaches:

  • Cost-sensitive learning — penalize misclassification of minority more
  • Ensemble methods — bagging with balanced bootstraps
  • Focal loss — down-weight easy examples, focus on hard ones

Use the right metrics: Not accuracy. Use F1, PR-AUC, recall, or domain-specific metrics.

▼ Click to expand
🔮

Data Augmentation

Synthetically increase your training set by applying transformations that preserve the label but change the input.

Computer vision: rotation, flipping, cropping, color jittering, CutMix, MixUp, random erasing.

NLP: synonym replacement, back-translation, random insertion/deletion, paraphrasing with LLMs.

Tabular: SMOTE, noise injection, feature perturbation. Less common but useful for rare events.

Key principle: augmentations must be label-preserving. Rotating a "6" image 180° makes it look like a "9" — that's bad augmentation.

▼ Click to expand
⚠️ Common Pitfalls
  • Label leakage — the label somehow gets encoded in the features (e.g., a "fraud" tag in the user profile used as a feature)
  • Survivorship bias — you only have data on approved loans, not rejected ones
  • Label ambiguity — different annotators disagree on 20% of labels, but you treat them as ground truth
  • Temporal leakage — using future information to predict the past in your train/test split

🔬 Dig Deeper: Data Quality & Data-Centric AI

+

The Data-Centric AI movement (championed by Andrew Ng) argues we should focus on systematically improving data rather than model architectures:

  • Consistent labeling — fix disagreements, create clear guidelines, measure inter-annotator agreement (Cohen's kappa)
  • Error analysis — systematically find slices where the model fails, then fix the data for those slices
  • Data cleaning — identify and fix mislabeled examples, remove duplicates, handle outliers
  • Data valuation — techniques like Data Shapley to quantify how much each training example contributes to model performance
  • Curriculum learning — order training examples from easy to hard

Explore further: Study Andrew Ng's Data-Centric AI Competition. Look into Cleanlab for automated data quality. Read about Data Shapley for understanding which training examples matter most.

Chapter 5 Feature Engineering

Features are the bridge between raw data and model understanding. Good features make simple models powerful; bad features make complex models useless. Feature engineering is where domain expertise meets ML craft.

Core Insight

A feature is a measurable property of your data presented in a form the model can use. The art is knowing which transformations surface the signal and which bury it in noise.

Common Feature Engineering Operations

🔢

Handling Missing Values

Missing data isn't just an inconvenience — the pattern of missingness itself can be informative.

Three types of missing data:

  • MCAR (Missing Completely At Random) — safe to drop/impute
  • MAR (Missing At Random) — missingness depends on observed data
  • MNAR (Missing Not At Random) — missingness depends on the missing value itself (most dangerous)

Strategies: deletion, mean/median/mode imputation, indicator variable for missingness, model-based imputation (KNN, iterative). Always create a binary is_missing feature — it's often predictive!

▼ Click to expand
📐

Scaling & Normalization

Features at different scales confuse many models. Normalizing ensures each feature contributes proportionally.

  • Min-Max — scales to [0, 1]. Preserves distribution shape. Sensitive to outliers.
  • Standard (Z-score) — mean=0, std=1. Works well for normally distributed features.
  • Robust scaling — uses median and IQR. Handles outliers gracefully.
  • Log transform — compress right-skewed distributions (income, page views).

Critical: Fit the scaler on training data only. Apply the same transformation to test/production data. This is a major source of training-serving skew.

▼ Click to expand
🏷️

Encoding Categoricals

Models need numbers. Converting categories to numbers without introducing false orderings is an art.

  • One-hot — binary column per category. Safe but high-dimensional. Works when cardinality is low (< 50).
  • Label encoding — integers. Only for ordinal features (low/medium/high) or tree-based models.
  • Target encoding — replace category with mean of the target. Powerful but prone to overfitting (use regularization).
  • Feature hashing — hash categories into fixed-size vector. Good for high cardinality (user IDs, URLs).
  • Embedding — learned dense vector. Best for very high cardinality with deep learning.
▼ Click to expand
✂️

Feature Crossing & Interaction

Sometimes the signal isn't in individual features but in combinations. Feature crossing captures these interactions.

A feature cross combines two or more features to capture their interaction. Example: age and income separately may be weak signals, but age × income (young + high income = tech worker) can be very predictive.

  • Manual crosses — domain knowledge-driven (city + cuisine = neighborhood preference)
  • Polynomial features — x², x×y, x×y×z. Explosive dimensionality growth.
  • Deep & Cross Network — learn crosses automatically (used at Google)

Caution: Feature crosses increase dimensionality exponentially. Use feature selection aggressively afterward.

▼ Click to expand
🛠️ Feature Stores in Practice

Why Feature Stores Matter

A feature store solves three problems at once:

  • Consistency — same feature definition used in training AND serving (no training-serving skew)
  • Reusability — team A computes "user_30d_purchase_count" once, team B reuses it
  • Point-in-time correctness — avoids temporal leakage by serving features as they were at prediction time

Tools: Feast (open-source), Tecton (managed), Hopsworks, Featureform. If you're building production ML, you'll eventually build or buy one.

🔬 Dig Deeper: Feature Selection & Importance

+

More features ≠ better model. Feature selection is about finding the minimal set that maximizes performance:

  • Filter methods — statistical tests independent of the model (correlation, mutual information, chi-squared)
  • Wrapper methods — use the model as a black box (forward selection, backward elimination, RFE)
  • Embedded methods — feature selection built into the model (L1 regularization, tree-based importance)
  • SHAP/LIME — model-agnostic explanation methods that also serve as feature importance

Practical tip: Start with feature importance from a gradient boosted tree (XGBoost/LightGBM). Use SHAP for more nuanced understanding. Remove features that are highly correlated with each other (keep the one with higher importance). Monitor feature importance drift in production.

Explore further: Study Boruta algorithm for feature selection. Read about Shapley values in game theory and their adaptation to ML feature importance via SHAP.

Chapter 6 Model Development & Offline Evaluation

Model selection is not about picking the fanciest architecture. It's about finding the simplest model that meets your requirements — balancing performance, latency, interpretability, and maintainability.

1

Start with a Baseline

Random prediction, majority class, or simple heuristic. You'd be surprised how often a simple rule-based system gets you 80% of the way there. Never skip the baseline.

2

Try Simple Models First

Logistic regression, decision trees, gradient boosted trees. These are fast to train, easy to debug, and often competitive. XGBoost/LightGBM dominate tabular data.

3

Move to Complex Models When Justified

Neural networks, transformers, ensemble methods. Only if simple models hit a clear performance ceiling AND you have enough data AND the latency/cost is acceptable.

4

Experiment Rigorously

Track everything: hyperparameters, data versions, random seeds, metrics per slice. Use experiment tracking tools (MLflow, W&B, Neptune). If you can't reproduce it, it doesn't count.

Core Insight

Ensembles almost always beat single models. But ensembles are harder to serve and maintain. The production trade-off: a single well-tuned model that ships fast vs. an ensemble that ships slow but performs better. Choose based on business impact.

Offline Evaluation — Getting It Right Before Production

📏

Evaluation Metrics

Choosing the right metric is as important as choosing the right model. The wrong metric optimizes the wrong thing.

Classification: Accuracy, Precision, Recall, F1, PR-AUC (for imbalanced), ROC-AUC, Log Loss

Regression: MSE, RMSE, MAE, MAPE, R²

Ranking: NDCG, MAP, MRR

The key question: What does a mistake cost?

  • High cost of false positives → optimize precision (spam filter being too aggressive)
  • High cost of false negatives → optimize recall (cancer screening missing actual cancer)
  • Care about both → use F1 or set threshold based on business cost matrix
▼ Click to expand
🔀

Data Splits & Cross-Validation

How you split data for evaluation can introduce subtle biases that make your model look better than it really is.

Train/Val/Test split — basic approach. Use test set only at the very end.

K-Fold CV — more robust for small datasets. Average across K evaluations.

Time-based split — CRITICAL for time-series. Train on past, validate on future. Never shuffle temporal data randomly.

Stratified split — maintain class distribution in each fold.

Group split — ensure all data from a single entity (user, session) is in the same fold. Prevents data leakage through correlated samples.

▼ Click to expand
🔍

Slice-Based Evaluation

Aggregate metrics hide problems. A model with 95% overall accuracy might have 60% accuracy on a critical subgroup.

Always evaluate on meaningful slices:

  • Demographic slices — age groups, geographies, languages
  • Behavioral slices — new vs returning users, high vs low activity
  • Input property slices — short vs long text, high vs low resolution images
  • Business-critical slices — premium users, high-value transactions

Use tools like SliceFinder or What-If Tool to discover underperforming slices automatically.

▼ Click to expand
🧠 Memory Anchor

"BSEC" — Baseline → Simple → Ensemble → Complex

Always progress from simple to complex. Each step up must justify its added complexity with measurable improvement. Document why each step was needed. Many production systems happily run on step 2.

🔬 Dig Deeper: Hyperparameter Tuning & AutoML

+
  • Grid search — exhaustive but expensive. Only for small search spaces.
  • Random search — surprisingly effective. Often finds good configs faster than grid search (Bergstra & Bengio, 2012).
  • Bayesian optimization (Optuna, Hyperopt) — models the objective function, intelligently chooses next hyperparameters to try. Best for expensive evaluations.
  • Successive halving / Hyperband — allocate more resources to promising configs, kill bad ones early. Very efficient.
  • AutoML (Auto-sklearn, H2O, Google AutoML) — automates model selection + hyperparameter tuning + feature engineering. Good for baselines, but understand what it's doing.
  • Neural Architecture Search (NAS) — auto-design neural network architectures. Computationally expensive but powerful.

Practical tip: Use Optuna with Bayesian optimization for most tuning tasks. Set a compute budget upfront. Log every trial — failed experiments are as informative as successful ones.

Explore further: Read about multi-fidelity optimization (evaluate cheap proxies first, refine on full data). Study the lottery ticket hypothesis for neural network pruning.

Online vs Batch Prediction

Online prediction serves one request at a time with low latency; batch prediction processes large volumes offline.

Online prediction:

  • Real-time, low latency (ms to seconds)
  • Triggered by user action
  • Examples: search ranking, fraud detection, autocomplete
  • Challenges: tail latency, cold start, feature serving

Batch prediction:

  • Run periodically, process large volumes
  • Store results for later retrieval
  • Examples: recommendations precomputed nightly, risk scores
  • Cheaper but stale — doesn't react to recent behavior

Hybrid: Pre-compute base predictions in batch, adjust in real-time. Common pattern for recommendation systems.

▼ Click to expand
🔀

Model Compression

Make models smaller and faster without losing much accuracy. Essential for latency-sensitive and edge deployments.

  • Knowledge distillation — train a small "student" model to mimic a large "teacher." BERT → DistilBERT is the classic example.
  • Pruning — remove weights/neurons that contribute little. Unstructured (individual weights) or structured (entire channels).
  • Quantization — reduce precision from FP32 to INT8 or even INT4. 2-4x speedup with minimal accuracy loss. ONNX Runtime, TensorRT support this.
  • Low-rank factorization — decompose weight matrices into products of smaller matrices.

These can be combined: distill → prune → quantize for maximum compression.

▼ Click to expand
☁️

Edge vs Cloud Deployment

Where the model runs (user's device vs cloud server) has profound implications for latency, privacy, and cost.

Cloud: powerful hardware, easy to update, centralized monitoring. But: network latency, privacy concerns, ongoing compute cost.

Edge: low latency (no network), works offline, data stays on device. But: limited compute, hard to update, diverse hardware, no server-side monitoring.

Hybrid: Simple/fast model on edge, complex model in cloud. On-device model handles 80% of cases, cloud model handles edge cases. Best of both worlds.

Tools: TensorFlow Lite, CoreML, ONNX Runtime Mobile, MLX (Apple Silicon).

▼ Click to expand
🚦

Deployment Strategies

How you roll out a new model is as important as the model itself. Wrong deployment can cause outages.

  • Shadow deployment — new model runs alongside old one, serves no traffic but logs predictions. Compare performance safely.
  • Canary — route 1-5% of traffic to new model. If metrics look good, gradually increase.
  • Blue/Green — two identical environments, switch traffic atomically. Fast rollback.
  • A/B testing — split traffic between old and new model, measure business impact statistically.
  • Multi-armed bandit — dynamically allocate more traffic to the better-performing model.

Golden rule: Always have a fast rollback plan. Never deploy Friday afternoon.

▼ Click to expand
⚠️ Training-Serving Skew

The #1 deployment bug. Features computed differently during training vs serving:

  • Using Python libraries in training but Java in serving — numeric differences
  • Feature lookups from different data sources (batch tables vs real-time cache)
  • Preprocessing order differences (normalize before or after encoding?)
  • Time zone differences between training data and live requests

Solution: Share feature computation code between training and serving. Or use a feature store that guarantees consistency.

🔬 Dig Deeper: ML Serving Architecture

+
  • Model server (TorchServe, TF Serving, Triton, BentoML) — serves model behind an API. Handles batching, versioning, health checks.
  • Feature serving — low-latency retrieval of pre-computed features. Redis, DynamoDB, or dedicated feature store.
  • Prediction cache — cache repeated predictions. Huge win if your input space is bounded.
  • Async prediction — for non-interactive use cases, queue predictions and process in background (Celery, SQS).
  • Microservice vs Monolith — ML model as a microservice enables independent scaling and deployment. But adds network overhead and operational complexity.

Explore further: Study Uber's Michelangelo, Facebook's FBLearner, Google's TFX. Each represents a different philosophy for production ML serving. Learn about model mesh — Kubernetes-native model serving at scale.

Chapter 8 Data Distribution Shifts & Monitoring

Your model was trained on past data but predicts future data. When the future looks different from the past, your model silently degrades. Monitoring distribution shifts is how you catch this before users notice.

Core Insight

All models degrade over time. The question isn't "will my model drift?" but "how quickly will it drift and how will I detect it?" Without monitoring, you're flying blind.

Types of Distribution Shifts

📊

Covariate Shift

The input distribution P(X) changes, but the relationship P(Y|X) stays the same. The most common type of shift.

Example: You trained a product recommendation model on desktop users. Mobile users have different browsing patterns (different X), but their preferences given what they browse (P(Y|X)) are the same.

Detection: Monitor input feature distributions. Statistical tests (KS test, PSI) on each feature.

Fix: Importance weighting (upweight training examples that look like production data), or retrain with new data distribution.

▼ Click to expand
🎯

Label Shift (Prior Shift)

The output distribution P(Y) changes, but P(X|Y) stays the same. The base rates of classes change.

Example: Your fraud detection model was trained when 1% of transactions were fraudulent. A new attack vector pushes fraud to 5%. Fraudulent transactions still look the same, but there are more of them.

Detection: Monitor prediction distribution. If the model suddenly predicts way more positives, the prior may have shifted.

Fix: Adjust decision thresholds, recalibrate the model, or retrain with data reflecting the new prior.

▼ Click to expand
🌀

Concept Drift

The relationship P(Y|X) itself changes. The same input should now get a different prediction. The hardest to detect.

Example: Pre-COVID, a model predicting restaurant foot traffic learned "Saturday + good weather = high traffic." During lockdowns, this relationship broke. Same inputs, completely different outputs.

Types:

  • Sudden — abrupt change (pandemic, regulation change)
  • Gradual — slow shift over weeks/months (cultural trends)
  • Recurring — seasonal patterns (holiday shopping)
  • Incremental — imperceptible daily changes that accumulate

Detection: Monitor model performance metrics directly. Track accuracy/precision/recall on labeled production data.

🛠️ What to Monitor in Production

The ML Monitoring Stack

  • Operational metrics — latency, throughput, error rates, CPU/memory
  • Model metrics — accuracy, precision, recall on labeled samples (if available)
  • Data metrics — feature distributions, missing value rates, cardinality changes
  • Prediction distribution — histogram of predicted scores over time
  • Business metrics — click-through rate, conversion, revenue (the ultimate ground truth)

🔬 Dig Deeper: Statistical Tests for Drift Detection

+
  • KS Test (Kolmogorov-Smirnov) — compares CDFs of two distributions. Good for continuous features. Sensitive to all types of distributional changes.
  • PSI (Population Stability Index) — measures how much a distribution has shifted. Common in credit scoring. PSI < 0.1 = no shift, 0.1-0.25 = moderate, > 0.25 = significant.
  • Chi-squared test — for categorical features. Tests if observed frequencies match expected frequencies.
  • MMD (Maximum Mean Discrepancy) — kernel-based test for multivariate distributions. Can detect shifts in feature interactions.
  • Page-Hinkley test — sequential test for detecting change points in a time series.

Practical tip: Don't set alerts on every feature — you'll drown in false alarms. Identify your 10-20 most important features and monitor those closely. Use a slower cadence (daily) for less critical features.

Chapter 9 Continual Learning & Testing in Production

The best ML systems don't just deploy and forget. They continuously learn from new data and use production as a testing ground. This chapter covers how to keep your model fresh and validate changes safely.

Core Insight

A model is a snapshot of the world at training time. The world keeps moving. Continual learning is how your model keeps up — but it introduces new risks like catastrophic forgetting and training instability.

🔄

Retraining Strategies

How often and how much to retrain depends on how fast your data shifts and how expensive retraining is.

Stateless retraining — retrain from scratch on all data. Simple and reproducible, but expensive and slow.

Stateful training (fine-tuning) — continue training from the previous checkpoint with new data. Faster, but can forget old patterns (catastrophic forgetting).

Frequency spectrum:

  • Annually — slow-changing domains (geology, astronomy)
  • Monthly/Weekly — most enterprise ML (churn, recommendations)
  • Daily — fast-changing domains (ads, trending content)
  • Real-time — streaming/online learning (stock trading, real-time personalization)

Trigger-based: Retrain when drift is detected, not on a fixed schedule. More efficient but requires good monitoring.

▼ Click to expand
🧪

Testing in Production

Offline metrics never tell the full story. Real validation happens in production with real users and real consequences.

A/B Testing — gold standard. Split traffic randomly, measure business metric difference. Requires statistical significance (usually 1-2 weeks of data).

Shadow testing — run new model in parallel, log predictions but don't serve them. Zero risk.

Canary release — serve new model to tiny fraction (1%). Monitor closely. Expand if healthy.

Interleaving — for ranking: interleave results from old and new model, measure which gets more clicks. More sample-efficient than A/B testing.

Bandits — dynamically allocate traffic to better-performing model. Minimizes regret during testing.

▼ Click to expand
🛡️

Safeguards for Continual Learning

Automated retraining without safeguards is a time bomb. You need guardrails at every step.

Data validation — check incoming data quality before it enters training. Schema validation, range checks, freshness checks.

Model validation — automated checks before deployment:

  • Performance on held-out test set ≥ threshold
  • Performance on critical slices ≥ threshold
  • No significant regression on any metric
  • Inference latency within budget

Staged rollout — shadow → canary → 10% → 50% → 100%, with automated rollback at each stage.

Human in the loop — for high-stakes domains, require human approval before production deployment.

▼ Click to expand
🧠 Memory Anchor

"Train, Validate, Gate, Deploy, Monitor — TVGDM"

The continual learning loop: Train on new data → Validate on test set + slices → Gate (pass quality checks?) → Deploy (staged rollout) → Monitor (watch metrics). If monitoring detects drift, the loop restarts.

🔬 Dig Deeper: Online Learning & Real-Time Adaptation

+

Online learning updates the model with each new example. Extremely powerful for fast-changing environments:

  • Use cases: Ad click prediction (Facebook's GBDT + logistic regression), real-time pricing, trending content recommendations.
  • Challenges: Catastrophic forgetting, training instability, adversarial manipulation (users can craft inputs that "poison" the model).
  • Algorithms: SGD on mini-batches, Follow-The-Regularized-Leader (FTRL), Vowpal Wabbit.
  • Replay buffers: Keep a buffer of past examples and mix them with new data during training. Prevents forgetting.
  • Elastic Weight Consolidation: Slow down learning for weights important to old tasks. Balances plasticity (learning new) vs. stability (remembering old).

Explore further: Study how Google and Meta implement real-time model updates for ads ranking. Look into continual learning research — particularly Progressive Neural Networks and PackNet. Read about concept drift-aware ensemble methods.

Pipeline Orchestration

ML workflows are DAGs of dependent steps. Orchestrators manage execution order, retries, scheduling, and dependencies.

Key tools:

Experiment Tracking & Model Registry

If you can't reproduce an experiment, you can't improve on it. Track everything, version everything.

Experiment tracking captures:

  • Hyperparameters and configuration
  • Data version / snapshot used
  • Code version (git commit)
  • Metrics, learning curves, artifacts
  • Environment (dependencies, hardware)

Model registry is the source of truth for model lifecycle:

  • Which model versions exist
  • Which is currently in staging/production
  • Who approved it, when
  • Linked experiment and data version

Tools: MLflow, Weights & Biases, Neptune, CometML, DVC.

🏗️

Compute & Resource Management

ML workloads are compute-hungry and bursty. Managing GPU resources, distributed training, and cost is a key infrastructure challenge.

Training compute:

  • Single GPU → Multi-GPU (data parallelism, model parallelism)
  • Distributed training (Horovod, DeepSpeed, FSDP)
  • Spot/preemptible instances for cost savings
  • Cloud (AWS/GCP/Azure) vs on-prem vs hybrid

Inference compute:

  • CPU for simple models (usually sufficient for gradient boosted trees)
  • GPU for neural networks, especially transformers
  • Specialized hardware (TPU, Inferentia, custom ASICs)
  • Auto-scaling based on traffic patterns

Cost optimization: Right-size instances, use spot pricing, batch inference where possible, cache predictions, optimize model size.

ML Platform Maturity Levels

Not every company needs a full ML platform on day one. Build incrementally based on your actual needs.

Level 0 — Manual: Jupyter notebooks, manual model export, no automation. Fine for experimentation, not for production.

Level 1 — ML Pipeline Automation: Automated training pipeline, basic monitoring, CI/CD for model deployment. This is where most companies should aim first.

Level 2 — CI/CD for ML: Automated testing, data validation, model validation, automated retraining triggers, feature store integration.

Level 3 — Full Platform: Self-service ML platform. Any team can build, deploy, and monitor models independently. Centralized governance, multi-tenant resource management.

Principle: Don't build Level 3 infrastructure for Level 0 problems. Match your tooling to your actual pain points.

📏 Rule of Thumb

"Buy/adopt before you build." Building your own ML infrastructure is tempting but incredibly expensive. Use managed services and open-source tools. Only build custom when off-the-shelf doesn't fit your exact requirements after honest evaluation.

🔬 Dig Deeper: Build vs Buy & Tool Selection

+

  • Integration — does it work with your existing stack? (cloud provider, orchestrator, storage)
  • Lock-in — can you migrate away? is the format open?
  • Community — active development? good documentation? real-world case studies?

Chapter 11 The Human Side of Machine Learning

ML systems are used by people and built by people. Responsible ML isn't an afterthought — it's a design requirement. Fairness, interpretability, and trust determine whether your system helps or harms.

Core Insight

An ML system can be highly accurate and deeply unfair at the same time. Fairness is not a single metric — it's a set of trade-offs that must be navigated with context, empathy, and domain expertise.

⚖️

Fairness & Bias

ML models can amplify societal biases present in training data. Understanding where bias enters and how to mitigate it is essential.

Sources of bias:

  • Historical bias — the world itself is biased; data reflects that
  • Representation bias — some groups are underrepresented in data
  • Measurement bias — features measure different things for different groups
  • Aggregation bias — one model for all subgroups when behavior differs
  • Evaluation bias — benchmark doesn't represent real usage

Fairness metrics: Demographic parity, equalized odds, calibration across groups, individual fairness. These metrics often conflict — you can't satisfy all simultaneously (Impossibility theorem).

Mitigation: Pre-processing (rebalance data), in-processing (constrained optimization), post-processing (adjust thresholds per group).

▼ Click to expand
🔍

Interpretability & Explainability

If you can't explain why your model made a decision, can you really trust it? Interpretability is key for debugging, trust, and compliance.

Intrinsically interpretable: Linear models, decision trees, rule lists. You can read the model itself.

Post-hoc explanation methods:

  • SHAP — Shapley value-based. Theoretically grounded, model-agnostic.
  • LIME — local linear approximation around each prediction.
  • Attention visualization — for transformers (but attention ≠ explanation)
  • Counterfactual explanations — "if you had income $5K higher, you'd be approved"
  • Concept-based (TCAV) — explain in human-meaningful concepts

When it matters most: Healthcare, finance, criminal justice, any high-stakes decision. Also critical for debugging — if you can't explain a wrong prediction, how will you fix it?

▼ Click to expand
🔒

Privacy & Security

ML models can leak training data. Privacy-preserving techniques let you build useful models while protecting individual data.

Threats:

  • Model inversion — reconstruct training data from model
  • Membership inference — determine if a sample was in training data
  • Data poisoning — corrupt training data to manipulate model
  • Model extraction — steal model by querying it repeatedly

Defenses:

  • Differential privacy — add calibrated noise during training
  • Federated learning — train on distributed data without centralizing it
  • Secure multi-party computation — multiple parties compute on combined data without sharing it
  • Data anonymization/pseudonymization
▼ Click to expand
📋

Model Cards & Documentation

Every model in production should have a "model card" — a standardized document explaining what it does, how it was built, and its limitations.

A model card should include:

  • Model name, version, owner, date
  • Intended use and out-of-scope uses
  • Training data description
  • Performance metrics (overall and per-slice)
  • Fairness evaluation across relevant groups
  • Known limitations and failure modes
  • Ethical considerations

Model cards bring accountability and transparency. They make it harder to deploy a model without understanding its implications. They also help new team members onboard quickly.

▼ Click to expand
🧠 Memory Anchor

"FIPE" — Fairness · Interpretability · Privacy · Ethics

The four pillars of responsible ML. Every production model should be evaluated across all four dimensions. A model that excels on accuracy but fails on fairness is not a good model — it's a liability.

🔬 Dig Deeper: Building an ML Ethics Practice

+
  • Pre-deployment review — establish a checklist before any model goes live. Include fairness analysis, privacy review, failure mode analysis.
  • Red teaming — have a team try to break or misuse your model before users do. Especially important for generative AI.
  • Feedback channels — give users a way to flag bad predictions. This is both ethically right and a free data quality signal.
  • Regular audits — schedule quarterly reviews of model performance across demographic slices. Automate slice-based reporting.
  • Incident response — have a plan for when things go wrong. Who gets paged? What's the rollback process? How do you communicate to users?

Organizational: Consider creating an ML ethics committee or review board. Include diverse perspectives — not just ML engineers, but ethicists, domain experts, and representatives of affected communities.
