Book Overview

The End-to-End Map

This book gives you a framework for adapting foundation models to real-world problems — from picking the right model, through evaluation, prompt engineering, RAG, agents, finetuning, dataset engineering, inference optimization, to production architecture.

The core philosophy: AI engineering is the discipline of building applications on top of readily available foundation models — not training from scratch. The book focuses on fundamentals that outlast any specific tool.

Three Core Adaptation Techniques

💬 Prompt Engineering

Craft instructions to steer model behavior without changing its weights. Cheapest and fastest. Always start here.

🔍 RAG

Connect the model to external knowledge sources so it can answer with up-to-date, domain-specific information.

🔧 Finetuning

Further train the model on your data to change its weights. Most expensive, but most powerful for behavior change.

Key Mental Model

Start simple, escalate as needed. Begin with prompt engineering. If the model lacks knowledge → add RAG. If the model lacks behavior → finetune. Real-world systems often combine all three.

Key Concepts

🔡 Language Model

Encodes statistical information about language. Predicts how likely a word is in context. Basic unit: token (character, word, or subword). Two types: masked (fills blanks, BERT) and autoregressive (predicts next, GPT).

📈 Self-Supervision

Labels inferred from input data itself — no human labeling. A single sentence yields multiple training samples. This is what let LLMs scale to billions of parameters.

🌐 Foundation Models

Work across multiple modalities (text, image, video, audio). General-purpose and can be adapted via prompt engineering, RAG, or finetuning.

⚡ Completion = Power

An autoregressive model is a "completion machine." Translation, summarization, classification, coding — all frameable as completion tasks.

Why This Matters For You

You don't need to train from scratch. Model-as-a-service means you build on top. New workflow: build product first → invest in data/models only after promise. Iteration speed is your advantage.

Use Cases

💻 Coding

Autocomplete, generation, debugging, documentation. GitHub Copilot pattern.

🎨 Image & Video

Marketing materials, art concepts, video editing. Stable Diffusion, DALL-E patterns.

✍ Writing

Emails, reports, marketing copy, contract analysis. Human reviews and refines.

🤖 Conversational Bots

Customer support, internal assistants, tutoring. Most visible AI application pattern.

📋 Data Organization

Classification, extraction, tagging, structuring unstructured data.

⚙ Workflow Automation

Agents that chain tasks, call APIs, operate autonomously across multi-step processes.

🎯 Use Case Evaluation
▼

Before building, ask: Should I build this? Evaluate whether the use case has enough value, whether failure is tolerable, and whether existing non-AI solutions are sufficient.

📐 Setting Expectations
▼

AI outputs are probabilistic, not deterministic. A model right 90% of the time is wrong 10% of the time — at scale, that's a lot of errors. Set realistic expectations with stakeholders.

📅 Milestone Planning
▼

Start with a demo/MVP using prompt engineering. Add RAG for domain knowledge. Evaluate if finetuning is needed. Plan for ongoing maintenance.

🧠 Chapter 1 Retention Checklist

  • Language models predict the next token. Self-supervision removed the data labeling bottleneck.
  • Foundation models are multimodal and general-purpose — not just text.
  • Completion is the fundamental primitive — nearly every task maps to it.
  • AI engineering = building on existing models, not training from scratch.
  • New workflow: Product → Model → Data (not the reverse).

Chapter 02

Understanding Foundation Models

How models are built: training data choices, the transformer architecture, model sizing, post-training alignment, and the crucial concept of sampling.

Training Data — The Fuel

Most models rely on Common Crawl — web scrapes of 2-3B pages/month. English is ~46%; languages like Punjabi, Swahili, Bengali are <0.1%. Models inherit web quality issues: clickbait, misinformation, language imbalance.

Practical Impact

Non-English is slower, more expensive, less accurate. Burmese needs ~10x more tokens than English → 10x cost. Always benchmark in your target language.

The Transformer Architecture

🏗 Why Transformer Won

Replaced seq2seq/RNNs. (1) Attention lets model reference any input token. (2) Input tokens processed in parallel. Massively faster.

🔑 Attention Mechanism

Query (Q), Key (K), Value (V) vectors. Q = "what am I looking for?" K = "what does each token offer?" V = "what info does it contain?" Score = relevance.

⚡ Prefill vs Decode

Prefill: All input tokens in parallel → fast. Decode: One output token at a time → sequential bottleneck. Drives many optimization techniques.

Model Size & Scaling Laws

Three numbers define scale: parameters, training tokens, FLOPs. Chinchilla: tokens ≈ 20× parameters. A 7B model should train on ~140B tokens.

Sampling — The Underrated Game-Changer

Author's Favorite Section

Sampling is how a model chooses output from all possibilities. It explains hallucinations, inconsistency, and creativity. Right sampling strategy boosts performance with almost zero effort.

🌡 Temperature

Low (0.0-0.3): Deterministic, safe. High (0.7-1.5): Creative, risky. Factual tasks → low. Creative → higher.

🎯 Top-k & Top-p

Top-k: Only k most likely tokens. Top-p: Smallest set whose cumulative probability > p. Top-p adapts dynamically.

🧪 Test-Time Compute

Generate multiple candidates, select the best. Best-of-N, chain-of-thought, tree-of-thought. More compute at inference → better results.

🧠 Chapter 2 Retention Checklist

  • Training data distribution determines model capabilities and biases.
  • Transformer = attention + parallelism. Prefill (parallel) → Decode (sequential).
  • Scaling law: tokens ≈ 20× parameters for compute-optimal training.
  • Post-training (SFT + RLHF/DPO) turns raw models into assistants.
  • Sampling controls creativity-accuracy tradeoff. AI is fundamentally probabilistic.
🔬 Dig Deeper
  • Vaswani et al., "Attention Is All You Need" (2017) — the transformer paper
  • Chinchilla paper — "Training Compute-Optimal Large Language Models"
  • Experiment with temperature and top-p on the same prompt to build intuition
  • Study MoE (Mixture of Experts) architecture — Mixtral, DeepSeek use it
Chapter 03

Evaluation Methodology

The biggest hurdle to production AI. Covers language modeling metrics, exact evaluation, similarity measures, embeddings, and AI-as-a-judge.

Why Evaluation Is Hard

🎓 Intelligence Paradox

Smarter models = harder to evaluate. Few can tell if a PhD-level math solution is wrong.

🌊 Open-Ended Outputs

Many valid answers exist. Can't check against a single ground truth.

⬛ Black Box Models

No access to internals. Can only observe outputs. Root cause analysis is extremely difficult.

📈 Benchmark Saturation

GLUE saturated in 1 year. MMLU being replaced by MMLU-Pro. Always playing catch-up.

✅ Exact Evaluation
▼

Works when there's a verifiable answer. Functional correctness — does code pass tests? Limitation: many real tasks don't have a single correct answer.

📏 Similarity Measures (BLEU, ROUGE, BERTScore)
▼

BLEU = n-gram precision (translation). ROUGE = n-gram recall (summarization). BERTScore = embedding similarity. Limitation: high similarity ≠ high quality.

⚖ AI as a Judge (LLM-as-Judge)
▼

Use a strong model to evaluate another. Growing fast because it's cheaper than human eval. Pitfalls: position bias, verbosity bias, self-enhancement bias. Best practice: multiple judges, randomize position, calibrate against humans.

🧠 Chapter 3 Retention Checklist

  • Perplexity = 2^(cross entropy). Lower = better language model.
  • Open-ended outputs make ground-truth comparison nearly impossible.
  • AI-as-judge is fast but biased — always calibrate against human eval.
  • Embeddings enable semantic similarity: compare meanings, not words.
  • No single metric captures everything. Use a portfolio of methods.
Chapter 04

Evaluate AI Systems

From methods to practice: model selection, benchmark navigation, and building a rigorous evaluation pipeline.

Evaluation Criteria

🎯 Domain Capability

Can the model handle your domain? Medical, legal, financial tasks need domain knowledge.

📝 Generation Quality

Coherence, factual accuracy, relevance, safety. Different apps weight these differently.

📐 Instruction Following

Does it follow format requirements? Output JSON? Stay in character? Most underrated criterion.

💰 Cost & Latency

A perfect model at $10/query or 30s is useless. Always evaluate cost-performance tradeoff.

Warning: Benchmark Pitfalls

Public benchmarks are contaminated — models may have trained on test data. Leaderboards are gamed. Always run your own evaluation on data relevant to your use case.

Design Your Evaluation Pipeline

Step 1: Evaluate ALL components, not just the model
▼

A RAG system has a retriever AND generator. If retrieval fails, generator can't help. Evaluate each component independently AND end-to-end.

Step 2: Create an evaluation guideline
▼

Write down what "good" looks like. Create rubrics with examples. Without this, evaluation is vibe-checking.

Step 3: Define methods and data
▼

Choose methods for each criterion. Curate test sets representing real usage — easy, hard, and adversarial cases.

🧠 Chapter 4 Retention Checklist

  • Always evaluate on your own data — never trust benchmarks alone.
  • Evaluate components independently AND end-to-end.
  • Write evaluation guidelines before you start evaluating.
  • Cost and latency are evaluation criteria, not afterthoughts.
Chapter 05

Prompt Engineering

The cheapest, fastest adaptation technique. In-context learning, system prompts, best practices, defensive prompting, and protection against jailbreaks.

A prompt has: (1) Task instruction — what to do. (2) Context — background info. (3) Examples — few-shot demonstrations. (4) Output format — structure.

0️⃣ Zero-Shot

Just the instruction. Relies on pre-trained knowledge. Works for simple, common tasks.

🔢 Few-Shot

Instruction + examples of input/output. Model learns the pattern. Dramatically improves structured tasks.

🖥 System Prompt

Sets persona, constraints, global behavior. Persists across conversation. Critical for production.

Best Practices

✍ Write clear, explicit instructions
▼

Instead of "Summarize this," say "Write a 3-sentence summary for a non-technical audience focusing on business impact." Ambiguity is the enemy.

📚 Provide sufficient context
▼

Don't assume the model knows your company, product, or domain. Include everything it needs or use RAG.

🧩 Break complex tasks into subtasks
▼

Chain simpler prompts: extract → classify → generate. Each step is easier to debug.

🤔 Give the model time to think
▼

Chain-of-thought: "think step by step." The model generates better answers when it works through steps first.

🔄 Iterate and version your prompts
▼

Treat prompts as code. Version control them. Test changes against eval suite. Keep a regression test set.

Defensive Prompt Engineering

Security: The Two Threats

Jailbreaking: Tricking a model into bypassing safety guardrails. Prompt Injection: Inserting malicious instructions into user input that override system prompt. This is the #1 security risk — treat all user input as untrusted.

🧠 Chapter 5 Retention Checklist

  • Start zero-shot. Add few-shot only if needed.
  • Be explicit, provide context, break into subtasks, let the model think.
  • System prompts set persistent behavior — invest time in them.
  • Version and test prompts like code.
  • Prompt injection is the SQL injection of AI — never trust user input.

📑 Term-Based (BM25)

Fast, no ML needed, great for exact matching. Always start here — baseline you must beat before embeddings.

🧮 Embedding-Based

Convert text to vectors, nearest neighbors. Captures semantic similarity. Requires embedding model + vector DB.

🔀 Hybrid Search

Combine term + embedding results. Best of both. Most production RAG uses hybrid + reranker.

Key optimizations: Chunking strategy (most impactful decision), query rewriting, reranking with cross-encoders, metadata filtering.

Agents

Definition

An agent perceives its environment and acts upon it. In AI: a model that uses tools (APIs, code, search) and does planning (break goals into steps) to accomplish tasks autonomously.

🔧 Tools

Functions the agent can call: search, code execution, APIs, databases. Turns text generator into an actor.

🗺 Planning

Breaking goals into steps. ReAct, Plan-then-Execute, Reflection. Planning is where agents most often fail.

🧠 Memory

Short-term: Current context (limited). Long-term: External persistent storage. Critical for multi-step tasks.

🧠 Chapter 6 Retention Checklist

  • RAG = Retriever + Generator. Start with BM25 before vector search.
  • Chunking strategy is the most impactful RAG design decision.
  • Hybrid search + reranking = production-grade retrieval.
  • Agents = model + tools + planning + memory.
  • Planning is the weakest link — evaluate and constrain carefully.
🔬 Dig Deeper
  • Build RAG with BM25 first, then add embeddings and measure improvement
  • Study the ReAct paper (Yao et al., 2022)
  • Experiment with different chunking strategies (fixed, semantic, recursive)
  • Explore vector DBs: Pinecone, Weaviate, Qdrant, Chroma, LanceDB
Chapter 07

Finetuning

When and how to further train a model. Memory math, quantization, LoRA, PEFT, model merging, and practical tactics.

When to Finetune (and When Not To)

✅ Reasons TO

Improve task-specific quality. Structured output formats. Bias mitigation. Reduce latency (bake examples into weights). Model distillation.

❌ Reasons NOT To

Degrades other tasks. Expensive (data, compute, talent). Better prompts or RAG may suffice. Don't own base model. Small improvements may not justify cost.

Decision Framework

Lacks knowledge → RAG. Lacks behavior → prompts first, then finetune. Too slow/expensive → distill into smaller finetuned model.

LoRA — The King of PEFT

Weight Matrix W
n × m
→
Decompose A × B
n×r, r×m
→

~0.01% params
→

🤔 Why does LoRA work?
▼

Pre-training compressed the model's "intrinsic dimension." Task-specific weight changes live in a much lower-dimensional subspace. LoRA exploits this. Analogy: pre-training builds highways; finetuning adds a few exits.

⚙ LoRA Config Tips
▼

Rank: Start 8-16. Diminishing returns beyond 16. Matrices: All attention (Q,K,V,O) with lower rank beats fewer matrices with higher rank. Adding feedforward layers helps. QLoRA: LoRA + 4-bit quant. 65B on single 48GB GPU.

Model Merging: Combine specialized finetuned models without additional training. SLERP, TIES, DARE methods. Combine a coding LoRA + writing LoRA into one multi-talented model.

🧠 Chapter 7 Retention Checklist

  • Try prompt engineering and RAG before finetuning.
  • LoRA decomposes weight updates — trains ~0.01% of parameters.
  • QLoRA = LoRA + 4-bit quantization. Large models on consumer hardware.
  • Finetuning on task A can degrade performance on tasks B, C, D.
  • Model merging combines specialized models without retraining.
Chapter 08

Dataset Engineering

Data is the foundation. How to curate, acquire, augment, synthesize, and process datasets.

The Golden Data Trio

💎 Quality

10K crafted instructions beat 100K noisy ones. LIMA: 1,000 curated examples rival GPT-4 in 43% of cases. Quality > quantity.

🌈 Coverage (Diversity)

Represent full range of production inputs. Simple and complex, edge cases, different formats. Quality AND diversity together outperform either alone.

📈 Quantity

Advanced base models need fewer examples. Plot a performance gain curve to estimate how many more examples you need.

Data Synthesis with AI

🤖 AI-Powered Synthesis Techniques
▼

Direct generation: Prompt a strong model to create input/output pairs.

Evol-Instruct: Iteratively make instructions more complex using AI.

Self-Instruct: Generate instructions, inputs, outputs all from the model.

Model Distillation: Train a small student model to mimic a large teacher. DistilBERT = 60% size, 97% performance.

  • Production data is the best training data — if you can use it.
  • AI-powered synthesis enables bootstrapping without real data.
  • Model distillation: train small to mimic large.
  • Always deduplicate and inspect before training.

Inference Optimization

Making models faster and cheaper. Metrics, accelerators, model-level and service-level optimization.

Key Metrics

MetricMeasuresWhy You Care
TTFTTime to first tokenUser perception of speed
TPOTTime per output tokenStreaming quality
ThroughputTokens/second servedCapacity & cost
Utilization% GPU capacity usedUnderutilized GPUs waste money

Optimization Techniques

📦 Quantization

FP32 → FP16 → INT8 → INT4. Each step halves memory, doubles speed. INT4 makes 70B fit on consumer GPUs.

🔑 KV Cache

Store key-value pairs so they're not recomputed per token. Essential for autoregressive generation. Can become enormous for long sequences.

⚡ Flash Attention

Reorders attention to minimize GPU memory transfers. 2-4x faster, no approximation. Now standard.

🧐 Continuous Batching

Insert new requests as slots free up. Dramatically improves throughput vs static batching. Used by vLLM, TGI.

🔮 Speculative Decoding

Small draft model proposes tokens, large model verifies in parallel. 2-3x speedup, no quality loss.

✂ Model Distillation

Train smaller model to mimic larger. Ultimate optimization: use a smaller model that's good enough.

🧠 Chapter 9 Retention Checklist

  • TTFT = user perception; throughput = cost.
  • Quantization is the single highest-impact optimization.
  • KV cache + Flash Attention + continuous batching = modern inference stack.
  • Speculative decoding: free speedup, no quality loss.
  • Ultimate optimization: distill to a smaller model that's good enough.
Chapter 10

AI Engineering Architecture & User Feedback

The culmination: production architecture in 5 progressive steps, plus monitoring, orchestration, and feedback loops.

The 5-Step Progressive Architecture

⊕ Enhance Context

Add RAG, tool use, file uploads. "Feature engineering for foundation models." Almost universally the first step.

⊕ Put in Guardrails

Input: PII masking, prompt injection detection. Output: Hallucination catching, format validation, toxicity filtering.

⊕ Router & Gateway

Router: Different models for different query types. Gateway: Unified API across providers with load balancing, logging.

⊕ Caches

Exact: Reuse for identical queries. Semantic: Reuse for similar queries. Cheapest latency optimization.

⊕ Agent Patterns

Multi-step reasoning, tool chaining, autonomous execution. Most powerful but most complex.

User Feedback — The Growth Engine

👍 Explicit Feedback (Thumbs up/down)
▼

Direct but sparse. Most users don't bother. Those who do skew negative. Valuable signal, never rely on it alone.

🔄 Implicit Feedback (User behavior)
▼

Did the user copy, edit heavily, regenerate, continue, or switch tools? More abundant and honest than ratings.

♻ Feedback → Training Data Loop
▼

The flywheel: positive interactions become training data. Negatives reveal failure modes. Your model improves at what your users care about. This is the moat.

🧠 Chapter 10 Retention Checklist

  • Build progressively: context → guardrails → routing → caching → agents.
  • Guardrails on BOTH input and output. PII masking for external APIs.
  • Routers save cost by sending simple queries to cheap models.
  • Semantic caching can dramatically reduce costs.
  • Monitor latency, cost, quality, and failures. Not optional.
  • Implicit user feedback is your most valuable data source.
🔬 Dig Deeper
  • Study gateways: LiteLLM, Portkey for multi-model orchestration
  • Implement a simple feedback loop: log user edits, use them to improve prompts
  • Explore guardrail frameworks: Guardrails AI, NeMo Guardrails, Lakera
  • Read about observability: LangSmith, Langfuse, Arize for tracing AI pipelines

The Complete Mental Model

Building production AI is iterative and progressive. Start with the simplest approach. Evaluate rigorously. Escalate complexity only when simpler approaches fail. Treat data, prompts, and evaluation as first-class engineering artifacts. Always close the feedback loop.
